## §2. Цвета (slides 81-82)

### Иерархия цветов (slide 81 — критично!)

> «**Зеленый цвет — акцентный.**
> **Графитового — больше.**
> Белый и серый — основные цвета фона.»

**Это инверсия наивного предположения!** Зелёный — НЕ доминанта, а **точечный акцент**. Графит (`#222222`) — основная доля. Белый/серый — фон.

### Соотношение цветов на слайде (целевое)
- **Белый/серый фон:** 60-70% площади
- **Графитовые элементы (текст, заголовки):** 20-30%
- **Зелёный (акценты):** 5-10%
- **Доп. цвета:** 0-5% (только в инфографике)

### Расширенная палитра (slide 81)

> «Расширенная гамма используется **только** для дифференциации внутри сложных макетов, для иллюстраций и инфографики. **Если для дифференциации не хватает оттенков зелёного.** При этом использовать можно только палитру второго и третьего ряда. Насыщенные цвета **НЕ** используются.»

→ Brand Guardian должен флагать использование первого ряда расширенной палитры (`#0063FF`, `#FF00FF`, `#FF4517`, `#FF0642`) на текстовых слайдах.

### Дополнительный цвет — правило сочетания (slide 82)

> «Дополнительные цвета всегда применяются **только вместе** с фирменным зелёным. Верный способ: фирменный зелёный + **1 доп. цвет**. Где применяем: инфографика, иллюстрации, рендеры, графика, KV.»

→ На одном слайде максимум: **зелёный + один доп. цвет**.
→ На текстовых слайдах доп. цветов **не использовать** вообще.

### Тёмные слайды (slide 22)

> «Тёмные слайды стоит использовать как **акцентные**, разбавляющие контент. **Общий тон презентации должен оставаться светлым.**»

→ Проверка ритма: если в презентации >40% тёмных слайдов — это нарушение.
→ Тёмный divider (donor 13/54) допустим как смена ритма, но не доминанта.

## §3. Гайд по оформлению схем (slides 62-66) — КРИТИЧНО

### Масштаб (slide 63)
- Все иконки на слайде — **одного размера**.
- Минимальный размер шрифта в схеме — заданный шаблоном (12pt Display Regular).
- Масштаб иконки изменять только пропорционально (отжать галочку «Сохранять пропорции»).

### Цвета схем (slide 64)
1. **Светло-серый + фирменный зелёный** — основа схемы (если содержание позволяет).
2. **Не использовать большое количество ярких цветов в схеме.** Расставлять акценты, а не превращать всё в радугу.
3. Цветные блоки уместны только когда:
   - Нужно выделить сущность
   - Развести по цвету разные группы
   - Схема состоит из небольшого количества элементов (**до 10**)
4. Цвета — только из палитры основного шаблона (см. §2).

### Блоки (slide 65)
- **Не искажайте пропорции иконок!**
- Оставляйте больше пространства внутри блоков, если позволяет место.
- Три типа блоков:
  - **Основные** (с иконкой/контентом)
  - **Блоки для схем** (разведение сущностей по цветам)
  - **Объединяющие** (группируют несколько блоков)

### Стрелки (slide 66) — КРИТИЧНО

> «Основной цвет стрелок — **нейтральный серый**.»
>
> «Толщина линий — **1 pt**.»
>
> «**Не используй основной зелёный для стрелок** — на экранах с плохой цветопередачей его не будет видно, лучше выбрать оттенок темнее.»

→ Это **частая ошибка**. Зелёные стрелки == нарушение бренда.
→ Дополнительные цвета стрелок — только когда нужно разделить связи по цветам (legend).

### Иконки (slide 83)

> «Ссылка на полный набор иконок: https://wiki.sbercloud.tech/pages/releaseview.action?pageId=362690983»

→ Иконки **только** из официального пака. Не использовать сторонние.

### Редактируемые диаграммы (правило v1.4 — добавлено по user feedback 2026-05-05)

> «Все графики и диаграммы должны быть **редактируемыми**.»

→ В скилле это `slide_type: "chart_pptx_native"` (через `pptx.chart.add_chart()` —
настоящая PowerPoint chart с привязанным датасетом). Пользователь жмёт ПКМ →
«Изменить данные» → открывается встроенный Excel и можно править цифры.

→ **НЕ использовать** `chart_native` (matplotlib → PNG) — кроме случаев когда
требуются эффекты, недоступные в PowerPoint native chart: vertical reference
lines с подписями, прозрачные overlapping areas с особой стилистикой осей.

→ **Header chart-слайда** — в стиле content slides шаблона: 20pt SemiBold CAPS,
top-left (35, 60). Не 32pt и не без CAPS — должен совпадать визуально с
заголовками donor 21/29.

→ **Палитра серий** — accent_idx = GREEN (один hero), остальные распределяются
по `NON_ACCENT_COLORS` (graphite, pastel blue, pastel yellow, pastel mint, gray).
GREEN никогда не дублируется.

→ Для процентных долей — `type: "area_100"` (ось Y автоматически 0%-100%).

## §4. KPI и большие цифры (slide 43)

### Структура KPI слайда
| Элемент | Размер | Положение |
|---|---|---|
| Title слайда | 21pt SemiBold | top |
| Большая цифра (3-х значное число) | **199pt** | hero |
| Малая цифра (2-х значное) | 44pt | corner |
| Знак "%" | 88pt SemiBold | рядом с цифрой |
| Описание (max 3 строки) | **12pt Regular** | под цифрой |

### Канонические правила
- **«Не более 3-х строк»** в описании (slide 43 явно прописано)
- 199pt — для **главной** цифры. На слайде максимум 1-2 цифры в 199pt (не все три).
- Знак "%" всегда меньше цифры (88pt vs 199pt).

### ⚠️ Ограничения по числу знаков (lesson v0.19)
**199pt frame в donor 43/44 вмещает максимум 2 значащих знака.**

| Что вмещается | Примеры |
|---|---|
| ✓ 1-значные | "5", "8" |
| ✓ 2-значные | "12", "84", "98", "30" |
| ✓ Диапазоны 2+2 | "12-25", "56-78" (как в эталоне slide 43) |
| ✓ Дробные 2-значные | "2,5", "9,9" |
| ❌ 3-значные | "245" обрезается до "24" |
| ❌ 4-значные | "1500" не помещается |

**Если контент = 3+ цифры:**
1. Округлить до 2-значного («~25%» вместо «245»)
2. Использовать единицы измерения («1,5K» вместо «1500»)
3. Сместить акцент на процент («24%» вместо «245»)
4. Использовать donor 47 (диаграмма с цифрой 80pt — больше места)

**Эталон slide 43:** числа `12, 34, 56-78` — все 2-значные. Layout Designer обязан проверять `len(num_*) ≤ 2` для donor 43/44.

## §4b. Таблицы (lesson v0.19)

Donor 53/54 («Большая таблица», «серая таблица») имеют в layout **brand-styled PNG-заглушку** с placeholder-таблицей («Столбец 1/Строка 1/+»). Это растровое изображение, не редактируемая таблица.

### Как использовать donor 53/54

| Сценарий | Что делать |
|---|---|
| Реальные tabular данные | Использовать **donor 35** (text-frame cells) ИЛИ donor 53/54 + `remove_before_fill` + add_table |
| Иллюстрация «есть таблица» (без деталей) | Donor 53/54 OK — placeholder показывает структуру |
| Сравнение 2-3 элементов | Donor 28 (2-col) или 34 (3-col) с подзаголовками |

### Plan для real table в donor 53/54

```json
{
  "clone_from_slide": 53,
  "remove_before_fill": [3, 4, 5],
  "table": {
    "rows": 4, "cols": 5,
    "headers": [...],
    "data": [...]
  }
}
```

**Layout Designer prompt:** при выборе donor 53/54 спрашивать «есть ли реальные данные?» — иначе использовать donor 35 или fallback к 2-col layout.

См. `memory/feedback_table_donor_needs_data.md`.

## §8. Word density (производное правило)

Из «не превращайте в текстовый документ» (slide 87 пункт 5) + структура слайдов 21-35:

| Тип слайда | Max слов |
|---|---|
| Title | 8-12 |
| Divider | 3-6 (название + номер) |
| 1-column body | 80 |
| 2-column body | 60 каждая |
| 3-column body | 35 каждая |
| 4-column body | 30 каждая |
| 6-blocks body | 20 каждый |
| 8-blocks body | 15 каждый |
| Callout (важная мысль) | 25 |
| KPI описание | 15 (3 строки × 5 слов) |

→ Если контент превышает — Overflow Strategy 2 (split на 2 слайда), не «утрамбовать».

### Split overcrowded slides (v1.4)

**Принцип:** перегруженный source slide → разбить на 2+ output slides. Это canonical поведение Cloud.ru дизайнеров (slide 87 пункт 5: «Не превращайте презентацию в текстовый документ»).

| Source имеет | Output |
|---|---|
| 4+ KPI цифр | 2 kpi_native (2+2 или 3+1) |
| 6+ блоков | 2 multicolumn (3+3 или 4+2) |
| Body > 80 слов на колонку | 2 content slides |
| 5+ images | 2 image_grid (3+2) |
| Chart с 5+ series | упростить до 2-3 + опц. split «context» + «detail» |
| Callout > 30 слов | 2 callouts |
| Title > 60 chars + body | title-slide + content-slide |

Логика split в `agents/02-slide-classifier.md` — Slide Classifier решает split до Layout Designer выбирает doors.

## §9. Применение в Layout Designer

При выборе donor LLM-агент `04-layout-designer.md` должен:

1. **Считать слова** контента → выбрать donor с подходящим word density (§8)
2. **Применить canonical размеры** при override (§1) — НЕ выдумывать свои
3. **Соблюдать иерархию цветов** (§2) — зелёный точечно, графит больше
4. **Для схем/диаграмм:**
   - Стрелки серые 1pt, **никогда не зелёные** (§3)
   - До 10 элементов — цветные блоки можно
   - Иконки одного размера, не искажать пропорции
5. **Для KPI:** размеры строго 199/88/12pt (§4)
6. **Команда:** выбирать donor по количеству людей (§5)
7. **Скриншоты:** только во фрейм-donor 72/73 (§6)

## §10. Применение в Brand Guardian

Расширения над текущей валидацией:

1. **Tone balance:** flag если >40% тёмных слайдов
2. **Зелёные стрелки на схемах:** flag любые pure-green линии в shape-типах connector/line
3. **KPI consistency:** на одном слайде максимум 2 цифры в 199pt
4. **Word density:** считать слова в text-frame и сравнивать с §8
5. **Доп. цвета без зелёного:** flag если на слайде есть Yellow/Purple/Blue без Green
6. **Расширенная палитра первого ряда (`#0063FF`, `#FF00FF`):** flag на текстовых слайдах

## §11. Применение в LLM 5-dim critique

К существующим 5 измерениям добавить чек-листы из §7 (10 приёмов):

- **Philosophy:** «Один слайд = одна мысль» — есть единая идея?
- **Hierarchy:** контраст между title/body/captions достаточен?
- **Detail:** есть ли висящие предлоги? Иконки одного размера?
- **Function:** «Рассказывайте историю» — есть ли narrative arc?
- **Innovation:** не превращён ли слайд в текстовый документ?

## §12. Зелёный — точечный акцент (правила применения)

Из slide 81: «Зеленый — акцентный».

**Правильно использовать зелёный:**
- Числа KPI (одна из трёх — зелёная)
- Подсветка ключевого слова в body (1 фраза, не весь параграф)
- Кнопка/CTA элемент
- Брендовые декоративные shapes из шаблона
- Логотип

**Неправильно использовать зелёный:**
- ❌ Заголовки слайдов целиком (= графит)
- ❌ Большие площади фона (= белый/серый)
- ❌ Все колонки 3-col одинаково зелёные (= dilution акцента)
- ❌ Стрелки в схемах (см. §3)
- ❌ Body-текст (= графит)
